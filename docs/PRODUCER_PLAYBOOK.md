# L5GN-Tools ‚Äî Producer Deployment & Operator's Playbook

How to stand up a **producer** rig, deposit its estate to the knight, and run it
day to day. Written for the work laptop, but the steps are the same for any
producer. Written to be handed to someone else.

> The **consumer** side ‚Äî standing up the knight, ingest, the read surface,
> backups ‚Äî is `KNIGHT_PLAYBOOK.md`. Read `ARCHITECTURE.md` ¬ß2 first if the
> producer/consumer split isn't familiar.

> **Why this matters right now.** The work rig has never deposited. Until it
> does, every MCF project exists in the registry as a link target with no repo
> facts attached ‚Äî `build_registry.py --report-aliases` shows them as
> `NOT IN ANY DEPOSIT`. This deploy is what unblocks the MCF half of the
> registry, and it is the first real exercise of the deposit wall.

---

## 1. What this is

A producer scans its own project folders into an `estate.json` snapshot and
**pushes** it to the knight. It does nothing else: it never consumes, never
ingests, never reads the vault.

**The wall, stated plainly.** A producer sends *only* what `run.py deposit`
stages ‚Äî the estate snapshot and its manifest. It never sends repo contents, and
the knight never reaches back onto this machine. Work and personal estates land
in separate directories on the knight and are never merged. If a step here looks
like it would copy source code off the work rig, stop ‚Äî it shouldn't.

**Which machine am I on?** This walkthrough crosses machines at step 9. Every
step is marked:

- **‚ñ∏ RIG** ‚Äî run on the producer you are standing up (PowerShell examples).
- **‚ñ∏ KNIGHT** ‚Äî run on the knight over SSH (bash examples).

## 3. Prerequisites ‚ñ∏ RIG

- **Python 3.10+** ‚Äî `python --version`.
- **git**.
- **SSH access to the knight**, ideally as an alias in `~/.ssh/config` so the
  push target reads `l5gn-castle:vault/estates` rather than a raw host string.
  Confirm with `ssh l5gn-castle 'echo ok'` ‚Äî it must print `ok` without
  prompting for a password.
- **This rig's hostname** ‚Äî `python -c "import socket; print(socket.gethostname())"`.
  **Write it down**; every config entry is keyed by it, and the key is
  case-sensitive.

## 4. Install ‚ñ∏ RIG

### 4a. Clone the toolkit

```powershell
cd C:\Users\<you>\Github          # wherever you keep repos
git clone https://github.com/<owner>/L5GN-Tools.git
cd L5GN-Tools
```

**Verify:** `git log --oneline -1` prints a commit.

### 4b. Create the virtual environment

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1      # Windows PowerShell
# source .venv/bin/activate       # macOS / Linux
```

**Verify:** `python -c "import sys; print(sys.prefix)"` prints a path inside
`.venv`. If it prints the system Python, the activate didn't take ‚Äî every step
below assumes it did.

### 4c. Install the toolkit

```powershell
pip install -e .
```

The core is stdlib-only, so this installs no third-party packages ‚Äî that is
expected, not a failure. A producer needs none of the optional extras
(`[viewer]`, `[review]`, `[chronicler]`, `[scrape]`); those are all consumer-side.

**Verify:**

```powershell
python -c "import l5gntools; print(l5gntools.__version__)"
```

It must print a version. This import is load-bearing beyond the toolkit's own
commands: `chronicler/pipeline/*.py` imports `l5gntools.dbsafe` for its
connection pragmas (DECISIONS 0014), so a pipeline script run outside this
environment fails with `ModuleNotFoundError: No module named 'l5gntools'`.

### 4d. Turn the gate on

```powershell
git config core.hooksPath .githooks
python verify.py
```

**Verify:** prints `verify: GREEN`. The pre-commit hook now refuses a red commit.

## 5. Configure this machine ‚ñ∏ RIG

Create `config\local.json` (git-ignored ‚Äî it never syncs to GitHub, which is the
point: employer paths and codenames stay off the public repo). Copy
`config\local.json.example` as a starting point and use **your hostname** from
¬ß3 as the key:

```json
{
  "RENAME-ME-WORK-LAPTOP": {
    "role": "producer",
    "estate": "work",
    "roots": [
      {"path": "D:/Work/Github/MCF",  "scope": "mcf"},
      {"path": "D:/Work/Github/L5GN", "scope": "l5gn"}
    ],
    "push_target": "l5gn-castle:vault/estates",
    "push_transport": "scp"
  }
}
```

Four things are load-bearing:

- **`role: producer`** ‚Äî this machine scans and pushes; it never consumes.
- **`estate: work`** ‚Äî the landing directory on the knight, and the wall
  boundary. It **must** be `work` here; `personal` is the gaming rig's.
- **`roots`** ‚Äî the folders whose *child* directories are the projects to scan.
  Point these at the `MCF/` and `L5GN/` folders that actually exist here.
- **`scope` on each root** ‚Äî how a project gets its registry scope (DECISIONS
  0012). Scope is declared on the root, never inferred from folder nesting,
  precisely so no machine has to reorganise its folders to match a convention.
  An untagged root still scans; its projects land as scope `other` and are listed
  under DEPOSIT GAPS by `build_registry.py --report-aliases`.

**Verify:**

```powershell
python run.py config
```

It must print your hostname with **no** `(no matching entry -> using default)`,
`role: producer`, `estate: work`, and each root listed *without* `(MISSING)`. A
`(MISSING)` root is a wrong path ‚Äî fix it now; a scan against a missing root
produces an empty estate, not an error.

## 6. First scan ‚ñ∏ RIG

```powershell
python run.py build
```

**Verify:** `data\estate.json` exists and lists your real projects:

```powershell
python -c "import json; d=json.load(open('data/estate.json')); print(len(d['projects']), [p['name'] for p in d['projects']][:10]); print(d['roots'])"
```

You should recognise the project names as your actual repos, and `roots` should
show each path with its scope tag. If `scope` is `null` on the projects, ¬ß5's
root tagging didn't take ‚Äî fix it before depositing. Scope travels with the
deposit, and a re-scan is cheap.

## 7. Stage the deposit ‚Äî no push yet ‚ñ∏ RIG

```powershell
python run.py deposit
```

Without `--push` this only *stages*, and prints the push command it would run.
Read that line before you send anything off the work machine.

**Verify:** `data\outbox\work\` contains `estate.json` and
`deposit_manifest.json` ‚Äî and nothing else you wouldn't want leaving the rig.

---

## 8. Push to the knight ‚ñ∏ RIG

```powershell
python run.py deposit --push
```

**Verify:** the output says `pushed : OK -> l5gn-castle:vault/estates/work/`.
Then, on the knight:

```bash
ls -la ~/vault/estates/work/
```

`estate.json` and `deposit_manifest.json`, with a current timestamp.

## 9. Consume ‚ñ∏ KNIGHT

You are now on the knight. The rig's part is done until ¬ß11.

```bash
cd ~/L5GN-Tools
.venv/bin/python run.py consume
```

**Verify:** the output lists the `work` estate with `manifest_verified: true` ‚Äî
the sha256 in the manifest matches the file that landed. A failed verify means a
truncated or tampered transfer; do not consume past it.

First run reports `estate_diff=insufficient_history`, because one snapshot cannot
be diffed. That is correct, not a fault; it becomes a real diff after a second
deposit on a later day.

## 10. Rebuild the registry with both estates ‚ñ∏ KNIGHT

The payoff ‚Äî the first time the registry can see the MCF projects.

```bash
cd ~/L5GN-Tools
.venv/bin/python chronicler/pipeline/build_registry.py --report-aliases
```

**Use the venv's interpreter explicitly.** These pipeline scripts run with the
pipeline directory on `sys.path`, not the repo root, so they resolve `l5gntools`
only when the toolkit is installed in the interpreter running them (¬ß4c). A bare
`python3` outside the venv fails at the import.

**Verify:** the ESTATE SOURCES block at the top lists **both** `personal` and
`work` with no `MISSING estate` line, and the MCF projects now carry real repo
facts (`present`, a `first_seen` date) instead of `NOT IN ANY DEPOSIT`.

When the list looks right, write it:

```bash
.venv/bin/python chronicler/pipeline/build_registry.py
```

> If the run reports `no such table: projects`, the **vault path** is wrong ‚Äî not
> the registry. Check `CHRONICLER_HOME` and `run.py config` before re-running: a
> mis-set path currently creates an empty `chronicler.db` at the wrong location
> rather than refusing, so the error names the wrong culprit.

## 11. Operator's guide ‚ñ∏ RIG

### Everyday commands

```powershell
python run.py build            # re-scan this rig
python run.py deposit --push   # ship the snapshot to the knight
```

Two commands, whenever you want the knight's picture refreshed. Everything else
is consumer-side. There is no schedule to keep ‚Äî the estate is a snapshot, and a
stale one is honestly stale rather than wrong.

### Reading what you sent

```powershell
python run.py config           # this machine's resolved role and paths
start report.html              # the human-readable scan of this estate
```

`report.html` and `data\` are regenerated output, git-ignored, and safe to
delete; `build` recreates them.

### Ground rules

- **Never commit `config\local.json` or `config\project_registry.json`.** Both
  are git-ignored and both carry employer paths or codenames. They ship by scp,
  not by git.
- **Never run `consume`, `ingest` or `review` here.** Those are consumer
  commands; this machine has no vault.
- **The gate applies here too.** `python verify.py` before a commit; the
  pre-commit hook enforces it.
- **A red gate is not a suggestion.** Fix it or don't commit.

## 12. Future: auto-apply on pull ‚Äî documented, NOT implemented

A `post-merge` git hook on the knight could run `verify.py` (and optionally
`consume`) automatically after every `git pull`, so a deploy is one push.

**Not implemented, deliberately.** A `post-merge` hook turns `git pull` into
*execute arbitrary code from the remote*. That is a genuine supply-chain surface:
anyone who can land a commit on the remote can run code on the knight, and `pull`
stops being the safe, inspectable operation it looks like.

The caveat is *acceptable here* because this is a repo only Tim controls, with no
third-party contributors and no CI bot with write access. It would not be
acceptable on a shared repo. If it is ever implemented, the conditions to
re-check are: still single-maintainer, still no automated writers to the remote,
and the hook runs `verify.py` **before** anything else, so a red gate stops the
deploy rather than deploying and then failing.

## 13. Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `ModuleNotFoundError: No module named 'l5gntools'` | a pipeline script run outside the venv | use `.venv/bin/python` explicitly (¬ß10), or activate the venv first |
| `run.py config` shows `(no matching entry -> using default)` | hostname key doesn't match | re-read the hostname from ¬ß3; JSON keys are case-sensitive |
| a root prints `(MISSING)` | wrong path in `roots` | fix the path; use forward slashes even on Windows |
| `estate.json` has 0 projects | roots point at the project folders themselves, not their parent | `roots` are the *parents* whose children are projects |
| projects have `scope: null` | roots left untagged | use the `{"path": ..., "scope": ...}` form (¬ß5) |
| push fails with a host error | ssh alias not resolving | test `ssh l5gn-castle 'echo ok'` first |
| `manifest_verified: false` on the knight | truncated or partial transfer | re-run `deposit --push`; do not consume an unverified bundle |
| `no such table: projects` | wrong vault path, not a registry fault | check `CHRONICLER_HOME`; see the note in ¬ß10 |
| `sys.prefix` shows the system Python | venv not activated in this shell | re-run the activate from ¬ß4b ‚Äî PowerShell does not inherit it between windows |
