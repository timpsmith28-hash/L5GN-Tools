# L5GN-Tools — Knight Deployment & Operator's Playbook

How to stand up the headless knight, wire push-to-deploy from a rig, run the
mesh, and operate it day to day. Written to be handed to someone else.

---

## 1. What this is

L5GN-Tools is a read-only toolkit that builds one picture of a code estate and
reconciles it against a chat-history vault (Chronicler). It runs on several
machines that share one git repo:

- **Producers** (the gaming rig, the work laptop) scan their own project folders
  into an `estate.json` snapshot and **push** it to the knight.
- **The consumer** (the *knight*, a headless box) receives every estate, holds
  them side by side but **walled** (personal vs work never merge), and runs the
  interpret layer over them plus the shared chat vault.

Each machine knows its own role from `config/machines.json` (keyed by hostname);
machine-specific paths and secrets live in `config/local.json` (git-ignored).

Nothing the toolkit does can modify a scanned repo or the vault: the scanners are
read-only and stdlib-only, enforced by a gate (`verify.py`) that the pre-commit
hook refuses to let you bypass.

## 2. Expected locations (the map)

### On the knight (`l5gn-castle`, Ubuntu, home = `/home/YOU`)

| Path | What lives there |
|---|---|
| `~/L5GN-Tools/` | the deployed toolkit (working tree) |
| `~/deploy/L5GN-Tools.git` | bare repo the rig pushes to (push-to-deploy) |
| `~/vault/chronicler.db` | the frozen Chronicler vault the knight reads |
| `~/vault/estates/` | **the landing zone** — deposits arrive here |
| `~/vault/estates/personal/` | gaming-rig deposit (estate.json, manifest, history/, reports/) |
| `~/vault/estates/work/` | work-laptop deposit (same shape) |
| `~/vault/estates/_shared/` | `vault_reader.json`, `project_trail.json` (one vault, shared) |
| `~/vault/estates/<estate>/history/` | accumulated snapshots — what `estate_diff` compares |
| `~/vault/estates/<estate>/reports/` | `estate_diff.json`, `drift.json` per estate |
| `~/L5GN-Tools/config/local.json` | the knight's own config (vault + estates_dir); git-ignored, survives deploys |

### On a producer rig (e.g. LucasGoonPC, Windows)

| Path | What lives there |
|---|---|
| `…\L5GN-Tools\` | the toolkit clone (synced via GitHub) |
| `…\L5GN-Tools\data\estate.json` | latest scan (regenerated by `build`) |
| `…\L5GN-Tools\data\history\` | this rig's own snapshot trail |
| `…\L5GN-Tools\data\outbox\<estate>\` | staged bundle waiting to be pushed |
| `…\L5GN-Tools\config\local.json` | this rig's `push_target` (git-ignored) |

### Chat-import inputs (producer side, wherever Chronicler runs)

These feed the vault; the knight only ever sees the finished `chronicler.db`.
Until Chronicler is folded into this repo (roadmap #16), they live under the
Chronicler project folder:

| Path | What lives there |
|---|---|
| `Chronicler/raw_claude_files/` | Claude export (`conversations.json`, `projects/`) |
| `Chronicler/raw_gemini_files/Takeout/.../My Activity.json` | Gemini Takeout (personal) |
| `Chronicler/scraped_gemini/` | Gemini share-link scrapes |
| `Chronicler/zip_downloads/archive/<source>/<account>/` | archived raw imports (never overwritten) |
| `Chronicler/vault_staging/<source>/<account>/<thread_id>.md` | Obsidian-browsable rendered view |
| `Chronicler/chronicler.db` | the SQLite vault → **copy this to the knight's `~/vault/`** |

> `data/`, `report.html`, and `config/local.json` are git-ignored on every
> machine — they are per-machine scratch, never committed.

## 3. Prerequisites (knight)

- Python 3.10+ (`python3 --version`) and Git (`git --version`).
- SSH from the rig already works (`ssh l5gn-castle` logs in — you have this).
- No third-party Python packages are needed for the toolkit itself (the scanners
  are stdlib-only). Chronicler ingestion has its own deps and is not on the knight yet.

## 4. Install on the knight

### 4a. First install (GitHub clone — get it running)

```bash
cd ~
git clone https://github.com/timpsmith28-hash/L5GN-Tools.git
mkdir -p ~/vault/estates
# copy the frozen vault across from a rig (run FROM the rig, or scp from the knight):
#   scp chronicler.db l5gn-castle:vault/chronicler.db
```

Create the knight's own config (git-ignored, survives every deploy). First find
this machine's hostname:

```bash
cd ~/L5GN-Tools
python3 run.py config          # note the 'hostname:' line
```

Then write `config/local.json`, replacing `KNIGHT-HOSTNAME` and `YOU`:

```json
{
  "KNIGHT-HOSTNAME": {
    "role": "consumer",
    "estate": "both",
    "vault": "/home/YOU/vault/chronicler.db",
    "estates_dir": "/home/YOU/vault/estates"
  }
}
```

Turn the gate on and confirm green:

```bash
git config core.hooksPath .githooks
python3 run.py config           # should now show role=consumer, vault, estates_dir
python3 verify.py               # verify: GREEN
```

### 4b. Push-to-deploy from the rig (the vertex3 pattern)

So future changes deploy straight from the gaming rig with one `git push`.

**On the knight**, create a bare repo whose `post-receive` hook checks out into
`~/L5GN-Tools` and runs the gate:

```bash
mkdir -p ~/deploy && cd ~/deploy
git init --bare L5GN-Tools.git
cat > L5GN-Tools.git/hooks/post-receive <<'EOF'
#!/bin/bash
set -e
DEPLOY="$HOME/L5GN-Tools"
BARE="$HOME/deploy/L5GN-Tools.git"
git --work-tree="$DEPLOY" --git-dir="$BARE" checkout -f main
cd "$DEPLOY"
echo "[deploy] running verify gate..."
python3 verify.py
echo "[deploy] now at $(git --git-dir="$BARE" rev-parse --short main)"
EOF
chmod +x L5GN-Tools.git/hooks/post-receive
```

`checkout -f` only touches tracked files, so `config/local.json`, `data/`, and
`~/vault/` are never disturbed by a deploy.

**On the gaming rig**, add the knight as a second remote and deploy:

```powershell
git remote add knight l5gn-castle:deploy/L5GN-Tools.git
git push knight main
```

From now on: `git push knight main` deploys and self-verifies on the knight. (You
still `git push` to GitHub as the canonical source; the knight remote is just the
fast path.)

## 5. First live round-trip

1. **Rig** — set the push target once in `…\L5GN-Tools\config\local.json`:
   ```json
   { "LucasGoonPC": { "push_target": "l5gn-castle:vault/estates", "push_transport": "scp" } }
   ```
2. **Rig** — scan, then push:
   ```powershell
   python run.py build
   python run.py deposit --push
   ```
   This `scp`s `data\outbox\personal\` to `~/vault/estates/personal/` on the knight.
3. **Knight** — ingest + interpret:
   ```bash
   cd ~/L5GN-Tools
   python3 run.py consume
   ```
   Reads each deposited estate, verifies its manifest, accumulates history, and
   writes `~/vault/estates/<estate>/reports/{estate_diff,drift}.json`.

First run shows `estate_diff=insufficient_history` (only one snapshot). After a
second deposit on a later day it becomes a real diff and `drift` starts reporting
`built_not_discussed`.

## 6. Scheduling (optional)

**Knight — nightly consume** (cron):
```bash
crontab -e
# every day at 06:00
0 6 * * * cd $HOME/L5GN-Tools && /usr/bin/python3 run.py consume >> $HOME/vault/consume.log 2>&1
```

**Rig — periodic deposit**: Windows Task Scheduler running
`python run.py build; python run.py deposit --push` (or use the desktop app's
scheduled tasks). Keep the cadence slower than or equal to the knight's, so each
consume has something new.

## 7. Operator's guide

### Mental model

Three verbs, by role:

- **build** (rig) — scan my projects → `estate.json` + a dated snapshot.
- **deposit** (rig) — package the latest snapshot and push it to the knight, into
  my own estate namespace only.
- **consume** (knight) — ingest every deposited estate and reconcile it against
  the chat vault.

### Everyday commands

| Command | Where | What it does |
|---|---|---|
| `python run.py config` | any | show this machine's resolved role/roots/vault |
| `python run.py list` | any | list every tool |
| `python run.py build` | rig | scan projects → `data/estate.json` + snapshot + `report.html` |
| `python run.py deposit` | rig | stage the namespaced bundle; print the push command |
| `python run.py deposit --push` | rig | actually push to the knight |
| `python run.py consume` | knight | ingest deposits + write per-estate reports |
| `python run.py <tool> --target NAME` | any | run one scanner on one project |
| `python verify.py` | any | run the gate (auditors + testers) |

### Reading the results

- **`report.html`** (rig, after `build`) — browsable estate dashboard.
- **`~/vault/estates/<estate>/reports/drift.json`** (knight) — the headline. Three signals:
  - `discussed_not_present` — chat talks about a project the knight holds in no
    estate. Usually means that repo lives on a machine that hasn't deposited yet.
  - `talked_not_built` — recent conversation, but the code hasn't moved.
  - `built_not_discussed` — new commits with no linked conversation.
- **`_shared/project_trail.json`** — per-project chat history, newest-first.

### Ground rules (for a collaborator)

- **Never commit red.** The pre-commit hook runs `verify.py`; a failing gate
  blocks the commit. Fix the gate, don't bypass it.
- **Scanners are read-only + stdlib-only.** A new scanner may import stdlib and
  `l5gntools` only, and must never write outside `data/`. The auditors enforce
  both. Writers (`deposit`, `consume`) live outside that contract on purpose.
- **The estate wall is structural.** A machine can only deposit into its own
  estate namespace; the knight keeps `personal` and `work` in separate trees.
  Don't cross them by hand.
- **`config/local.json` and `data/` are git-ignored** — never commit machine
  paths or secrets; put them there.
- **Add a scanner** by writing `l5gntools/scanners/<name>.py` (with `NAME`,
  `DESCRIPTION`, `ESTATE_LEVEL`, `SAFETY`, and `scan`/`scan_estate`) and importing
  it in `l5gntools/registry.py`. Nothing else needs to change.

### Troubleshooting

| Symptom | Cause / fix |
|---|---|
| `vault_reader … no_vault` | knight `vault` path wrong or `chronicler.db` missing — check `config/local.json` and `~/vault/`. |
| `… schema_mismatch` | the vault isn't the frozen one (`user_version` ≠ 1) — re-finalize it and re-copy. |
| `estate_diff … insufficient_history` | only one snapshot yet — normal until the second deposit. |
| `deposit: refusing … 'unknown' estate` | this machine has no `estate` set — add `estate: personal|work` to its config. |
| push fails / `rsync: command not found` | on Windows use `scp` (auto-picked) or set `"push_transport": "scp"`. |
| deploy `verify` fails on the knight | the pushed commit is red; fix on the rig and re-push (the rig's pre-commit should have caught it). |

## 8. Glossary

- **Estate** — one machine's set of project repos, snapshotted as `estate.json`.
- **Vault** — the frozen `chronicler.db`: the source-of-truth chat archive.
- **Deposit** — a namespaced bundle (estate.json + snapshot + manifest) pushed to the knight.
- **Knight** — the headless consumer that holds every estate walled and interprets them.
- **Drift** — the reconciliation of what was *built* against what was *discussed*.
